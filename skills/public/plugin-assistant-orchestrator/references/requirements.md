# Requirements Playbook

适用时机：用户要新建插件，或现有插件目标不清，需要把聊天内容收束成可生成的插件需求。

## 必问最小集
- 这个插件帮谁解决什么问题。
- 它从哪里进入：右键文件、右键目录、全部内容，还是固定测试资料。
- 用户完成一次任务的最短路径是什么。
- 成功标准是什么：节省时间、减少手工步骤、提升预览质量、批量处理等。

## 输出格式
- 目标：一句话说明插件价值。
- 目标用户：角色 + 使用场景。
- 插件边界：做什么，不做什么。
- 成功标准：1-3 条可验证结果。

## 收敛规则
- 缺信息时一次只追问 1-2 个高价值问题。
- 如果用户表达很散，先帮他总结再确认，不直接进入 UI。
- 如果是调试已有插件，默认沿用当前 manifest 能力边界，只问这次要改什么。
