# 产物中心搜索/筛选/排序设计（MVP+）

## 背景
当前产物中心已支持分组、拖拽移动、工作台打开和分组下载。随着产物数量增加，用户在抽屉内定位目标文件的成本明显上升，因此需要补齐轻量的检索能力。

## 目标
1. 在不改后端数据结构的前提下，提供客户端实时搜索、类型筛选与排序。
2. 与现有分组体系兼容，不破坏已保存分组和自动分组逻辑。
3. 保持操作成本低，适合高频使用。

## 范围
1. 搜索：按文件名和路径模糊匹配。
2. 筛选：全部、文档、图片、音视频、代码、技能包、其他。
3. 排序：最近生成、名称 A-Z、名称 Z-A。
4. 空状态：当有产物但当前条件无匹配时，显示“无匹配结果”提示。

## 实现方案
1. 新增 `frontend/src/core/artifacts/filtering.ts`，统一封装分类、过滤、排序逻辑：
   - `getArtifactFilterType(filepath)`
   - `filterAndSortArtifacts(files, { query, filterType, sortMode })`
2. `ArtifactCenter` 新增控制区：
   - 搜索输入框（`Input`）
   - 类型筛选（`Select`）
   - 排序（`Select`）
3. 渲染流程调整：
   - 先对全量产物执行过滤排序
   - 再映射分组并过滤空组
   - 最后计算未分组区
4. i18n 扩展：
   - `artifactCenter` 新增搜索/筛选/排序与“无匹配”文案（中英）。

## 关键取舍
1. 不增加后端查询参数：MVP 阶段由前端本地处理，减少接口复杂度。
2. “最近生成”保持现有产物顺序：当前没有稳定的单文件时间戳字段，避免伪造时间导致排序误导。
3. 分类依赖扩展名：实现简单且足够可用；后续可升级为 `metadata.type` 优先。

## 后续迭代建议
1. 增加“清空筛选”一键操作与结果计数。
2. 引入按文件大小/时间的高级排序（需后端提供元数据）。
3. 大列表虚拟滚动，降低产物量较大时的渲染开销。
