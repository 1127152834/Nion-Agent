# 2026-03-13 结论：主动性与注意力管理产品线

## 1. 本轮结论（一句话）

Nion 的下一阶段增长点应是“主动决策与注意力治理”，而不是继续堆叠执行能力。

## 2. 证据基础

### 2.1 仓库内能力现状

- 已有调度底座：`backend/src/scheduler/models.py`、`workflow.py`、`routers/scheduler.py`
- 已有通知机制：`frontend/src/core/notification/hooks.ts`
- 已有提醒监听：`frontend/src/components/workspace/scheduler/scheduler-reminder-watcher.tsx`
- 已有任务与子任务上下文：`TodoList`、`SubtasksProvider`

结论：缺的是“产品化的主动层”，不是“执行引擎本身”。

### 2.2 外部标杆信号（本轮调研）

- OpenAI Operator（任务执行型助手）
- Anthropic Claude 文档（extended thinking / MCP / tool use）
- Cursor Background Agent（后台异步代理）
- Cognition Devin（长时任务连续执行）
- GitHub Copilot Coding Agent（Issue 到 PR 的异步闭环）

共同趋势：从“问答助手”转向“持续行动代理”。

## 3. 产品方案（新增模块）

### 模块 A：Action Inbox

目标：让用户先看到“下一步行动”，而不是先回看长对话。

MVP 范围：
- 输入源：聊天待办、scheduler 任务、代理完成回执、失败重试
- 字段：title、priority、due_at、source、status、recommended_action
- 交互：接受建议 / 延后 / 忽略 / 打开上下文

### 模块 B：Priority Engine

目标：把“重要但不紧急”的事项从噪音中拉出来。

规则打分（可解释）：
- impact_score（影响）
- urgency_score（紧急）
- dependency_score（阻塞关系）
- context_switch_cost（切换成本）

输出：Now、Next、Later 三层队列。

### 模块 C：Interruption Recovery

目标：任何中断后都能 30 秒恢复上下文。

核心能力：
- 自动生成 `resume capsule`
- 记录中断点、最后成功步骤、失败原因
- 一键恢复执行或改道

### 模块 D：Opportunity Discovery

目标：系统主动发现“低风险高收益”的可推进事项。

例子：
- 昨日失败重试在依赖已满足后重启
- 定时汇总可自动完成并在收件箱待确认

### 模块 E：Attention Budget

目标：控制提醒总量，保护专注。

机制：
- 每小时主动提醒上限
- Focus Session 下仅高优先级提醒
- 低优先级合并成 digest

## 4. 落地建议（工程可执行）

1. 先建统一 Action 数据结构（可落 SQLite 或线程状态聚合层），不要先做复杂 AI 策略。
2. 优先做可解释规则引擎，再迭代模型策略，避免黑盒优先级。
3. 所有主动动作默认“建议优先、自动执行可配置且可追溯”。
4. 把现有 scheduler 历史与 notification 打通到一个收件箱视图，减少信息分散。

## 5. 风险与边界

- 风险 1：过度提醒导致用户反感
- 风险 2：自动执行越权
- 风险 3：优先级规则不透明

边界原则：
- 默认不静默执行高风险操作
- 任何自动动作必须可回放、可撤销、可审计
- 提醒频控是强约束，不是优化项

## 6. 下轮建议探索（与本轮不重复）

建议下轮切到“多设备连续体验与信任边界”：桌面/Web/IM 桥接的状态连续、权限继承、身份确认与安全提示设计。
