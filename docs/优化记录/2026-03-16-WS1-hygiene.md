## 背景

本仓库历史上进行过多轮快速试验与迭代，形成了较多“功能闭环存在但噪声与冗余积累”的现象：过期原型、重复报告、临时脚本、未再使用的静态产物等。这类内容虽然不一定直接导致运行时故障，但会显著抬高：

- 新人理解成本（不知道什么是当前主路径）
- 维护成本（不敢删、不敢改）
- 变更风险（误把过期代码当作现行实现）

因此 WS1 的目标是：先建立一份 **可审计的候选清单**，并对“可能删除的项”逐条补齐 **证据链**，再按小步、可回滚方式执行第一批低风险清理。

## 本阶段策略与约束

- 只从 D/C 级目录入手：`reports/`、`openspec/`、`docs/`（排除 `docs/plans/`、`docs/product-design/`、`docs/优化记录/`）、低风险脚本与静态产物。
- 删除必须有证据链：静态引用扫描 + 动态引用排查；不确定则先 `FREEZE`。
- 小步提交：一次只删 1-3 个最确定项，并保持可 `git revert` 回滚。
- 不做目录大搬家，不触碰 A 级核心域（Memory/Sandbox/Channels/Scheduler/对外 API 契约等）。

## 删除/迁移/冻结证据链（若适用）

### 术语

- `DELETE`：证据链充分，可直接删除（低风险）。
- `FREEZE`：不确定是否动态引用，先冻结/隔离/加告警，再观察。
- `MIGRATE`：先迁移/合并到新位置后，再删除旧路径。
- `KEEP`：保留（说明理由）。

### 启发式线索（vulture，仅作为候选输入，不可直接删除）

- Run: `cd backend && make deadcode`
- Result: vulture 返回 1 条候选（命令 exit code != 0 属正常现象）
  - `src/tools/builtins/a2ui_tool.py:14: unused variable 'a2ui_json' (100% confidence)`
- 结论（当前阶段）：判定为 **预期行为/非死代码**。
  - 原因：`send_a2ui_json_to_client_tool(a2ui_json=...)` 的入参是工具协议的一部分（供 LLM 传参生成 schema），即使 fallback 实现不使用该入参，也不应删除/改名参数，否则可能破坏 tool 调用契约。
