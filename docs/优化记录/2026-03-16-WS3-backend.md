## 背景

backend 在多轮快速迭代后出现明显的“热点超大文件”与“入口变厚”的问题：router/service 文件动辄上千行，逻辑与 IO/副作用混杂，导致：

- 变更时爆炸半径难评估（缺少清晰边界与可测的纯逻辑层）
- review 成本高（一个文件承担过多职责）
- 回归风险上升（局部修改影响全局，且难以快速定位）

WS3 的目标不是做“大爆炸重构”，而是在门禁与回滚机制完备的前提下，优先治理 B 级热点：**先盘点与固化证据，再做小步内部抽取与薄化入口**。

## 本阶段策略与约束

- Scope Freeze：
  - 不改变 Gateway 对外契约（路由路径、请求/响应结构、错误语义保持不变）。
  - 不做 A 级核心域的大手术（Memory/OpenViking、Sandbox、Channels、Scheduler）。
  - 不引入重型依赖或新抽象层（保持简单与可 debug）。
- 方法：
  - 先用现有测试（或补 characterization tests）钉住行为，再做内部抽取。
  - 每个提交只做一类事，保证可审查、可回滚（`git revert <sha>`）。
  - 合并前必须提供门禁证据（至少 `make verify-backend`；高风险改动再跑全栈 `make verify`）。
